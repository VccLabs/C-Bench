Giraffe IDE Introduction
Giraffe IDE is a human-machine interaction UI development tool developed by Shanghai Top Display Optoelectronics Co., Ltd.

The user can quickly implement a WYSIWYG(What You See Is What You Get) interface similar to the Android UI components by dragging and dropping the built-in controls with the mouse on the PC host software. This greatly simplifies the development process, shortens the development process, and improves the user experience.

GiraffeIDE's System Advantages:
Provides an internal system simulation window.

Provides a standard C language development compilation environment.

Own UI development framework.

Provides WYSIWYG configurable UI development tools.

Provides a complete open writing section for logic code.

Provides different control themes for more comfortable design.

Provides a rich API interface for users to call directly.

Remote update system capability.

Continuously updated control packages and system updates.

Continuously updated documentation and demo support.

What effects can Giraffe achieve?
Background size can be set to any size and can display any pixel image, if the image is larger than the view pixel, it will be automatically compressed.

All controls can be individually styled to ensure that each control is unique.

Label text control: It can display text in any font

Button control: As a general purpose input/output control, buttons are used extensively in UI design. The button text can be set directly in the button, eliminating the need to add a separate text control. The size and position of the button can be adjusted directly with the mouse, and button styles such as colour, rounded corners and borders can be set in the properties. Once the button is generated you can add your own code to the corresponding event function.

image control: you can add any image in .jpg or .png format.

Container control: the container control can be used as a carrier for any other type of control.

Image button control: you can set a different picture for each state of the button.

Line control: you can connect lines between coordinate points.

Arc control: control that can be used to draw arcs.

Bar control: can be used to display progress information.

Slider control: change the progress of the bar by sliding the slider.

SW control: change the state of the switch by sliding or clicking.

CB control: checkboxes can be checked/unchecked.

Mbox control: for message prompting, action selection and guiding the user through actions.

Txtbox control: the usual text input box with password mode.

List control: list item buttons arranged from top to bottom, with images and text inside the buttons.

Droplist control: a list control that can be collapsed and expanded.

Rollist control: a scrolling list control.

Chart control: display line, bar, scatter and area charts.

Table control: free to set the rows and content of the table control.

Spinbox control: Adjust data increment and decrement.

gif control:You can add .gif animated images.

QRcode control: Can be used to add QR codes containing specific information.

video control: For playing video files in .mp4 format.

keyboard control: You can enter Chinese and English text in the txtbox control.

2.12.2 Differences between Private Controls and Public Controls

In the properties on the right panel of a control, the Global control type option can be set to Private or Public. The differences between the two are as follows:

Public: Accessible across views and valid throughout the entire project.

By default, global controls are only loaded into memory when their display is switched to, and subsequent accesses read directly from memory; they are not released when exiting the view. If there is a need to access a control before it is displayed, you can call the grf_view_load function in the grf_main function to load the global controls of that view into memory. You can also modify the project property Initialize all views on power-up, but this will slow down the power-up startup time.

Private: Only valid for access when the current view is displayed, and released after switching views.

3.3.2 Possible Causes and Troubleshooting Steps
Physical connection issue
Cause: Damaged or loose USB data cable; or use of a charge-only cable that does not support data transfer.

Troubleshooting steps: Replace with a USB cable that supports data transfer. Reconnect the device and the computer. Check if the USB port on the device side is loose or damaged. Unplug and reconnect to ensure a good connection. Check if the power indicator is on to confirm the device is powered normally.

Driver issue
Cause: The device USB driver is not installed or not installed correctly on the computer.

Troubleshooting steps: Open Device Manager on the computer (Right-click This PC → Manage → Device Manager). Check for new devices under Universal Serial Bus controllers or Other devices.

Under normal conditions, shortly after connecting the device to the computer, FunctionFS gadget(adb) or ADB Interface will appear.

3. View Introduction
A view is a UI display page on which we can create various widgets, insert background images, set the location of the view display layer, adjust transparency, and so on. Each view has a corresponding name and id that we can call according to our needs to switch between different views.

As shown in the diagram:

img

The area indicated by the arrow is the view interface, and on the right are the view's property settings.

img

View Name: the name of the current view

View ID: ID of the current view

BackImage: add a background image for the view, the image supports both jpg and png formats, For details, please refer to Chapter 8.3 of the Software API

BackColor: the color of the view's background

BackGround Alpha: the transparency of the view background, in the range [0-255] 0 is completely transparent 255 is completely opaque

Rotation Display: Rotate a single view by a specified angle (0°, 90°, 180°, 270°).

Note: This attribute is used to meet the requirements of both landscape and portrait display within the same project, supporting clockwise rotation at a specified angle for individual views.Rotation of a single layer is not supported. The final rotation angle is based on the UI layer; rotation of other layers follows the UI layer. If you need to achieve the overall project display effect of "using landscape screen in portrait mode" or "using portrait screen in landscape mode", please use the screen rotation angle in the project properties: this property will rotate all layers and content of the entire project.

View's Display Layer Location: 4 types
GRF_LAYER_BG: Background Layer

GRF_LAYER_UI: UI layer.

GRF_LAYER_TOP: top layer.

GRF_LAYER_STATE: state layer.

These four layers are stacked from bottom to top in the priority order of GRF_LAYER_BG → GRF_LAYER_UI → GRF_LAYER_STATE → GRF_LAYER_TOP. The background layer acts as the bottom-most canvas, while the top layer is used for displaying highest-priority content such as pop-ups.

img

Example: When a control needs to be reused across multiple pages (with duplicate functions), the UI page to which the control belongs can be assigned to the state layer or top layer (note that the background transparency of this page must be set to 0). Then, call the API interface s32 grf_view_set_dis_view(u32 v_id); in the entry function of the target page where the function is required (e.g., void view1_entry(void)), and the overlay display effect of different layers can be achieved.

The view resources are loaded when the view is first displayed, switching views and re-entering will keep the original running state, and will not load the resources repeatedly. User can also call grf_view_load(u32 v_num) to load the view resources.

3.1. API
3.1.1. Set the display layer position
s32 grf_view_set_layer(u32 v_id,grf_layer_se layer); 
v_id: view id

Layer: The view layer to be displayed. The layers are divided into Background Layer, UI Layer, State Layer, and Top Layer, which correspond to GRF_LAYER_BG、GRF_LAYER_UI、GRF_LAYER_STATE、GRF_LAYER_TOP。

typedef enum 
{
    GRF_LAYER_BG,        //Background Layer

    GRF_LAYER_UI,        //UI layer

    GRF_LAYER_STATE,     //state layer

    GRF_LAYER_TOP,       //Top Layer
}grf_layer_se;
Return value: none

This interface can be used to switch display views

3.1.2. Set display view
s32  grf_view_set_dis_view(u32 v_id);
v_id: view id

Return value: none

This interface can be used to switch display views

3.1.3. Set display view
s32 grf_view_set_dis_view_anim(u32 v_id,grf_scr_load_anim_t anim_type, u32 time, u32 delay,grf_anim_path_e path_e);
v_id: view id

anim_type: Preset animation type

time: Animation duration delay

4. Widgets Introduction
4.1. Public Properties
4.1.1. Introduction to Basic Properties
Basic properties are common to all widget object, such as widget name, widget ID, location, touch mode, whether to hide, transparency, alignment, click setting, style, etc.

4.1.2. Basic Properties
The public properties include the widget name, widget ID, widget location(top left corner, widget width and height), touch mode, scroll bar display mode, whether to hide, transparency, etc.

img

Name: The name of the control, which can be modified. It must be unique within the same view, use only English (starting with a letter), and contain no spaces or special characters. It is not case-sensitive.
Control ID: The unique identifier of the control, which can be modified. It must be unique within the same view.
Global control type: The accessibility type of the control.
Global: Accessible across views and valid for the entire project. By default, global controls are loaded into memory only when switching displays. Subsequent accesses read directly from memory, and they are not released when exiting the view. If you need to access the control before it is displayed, you can call the grf_view_load function in the grf_main function to load the global controls of this view into memory. You can also modify the Initialize all views on power-on option in the project properties, but this will slow down the power-on startup time.
Private: Accessible only when the current view is displayed. It is released after switching views.
Ctrol Id Enum: View enumeration identifier
Location: The position relative to the parent control, including the top-left coordinates, width, and height.
Touch Mode: Whether the control responds to touch events. It supports clicking by default.
Expanded Touch range: Expand the touch area outward (up, down, left, right) relative to the control size. The unit is pixels.
To Select Events:Used to define the types of events that controls can trigger and report. By checking different event options, you can control the control’s response to operations such as touch, click, scroll, value change, etc. The purpose of this configuration item is to check only required events, avoid unnecessary event responses, and improve system operating efficiency.
Rolling Mode: Only valid for container-type controls. It configures the display and operation mode when the displayable content exceeds the control size.
Scroll bar display mode: The display mode of the rolling bar.
Hidden: The scrollbar is not displayed regardless of whether the content overflows the viewable area.
Always Show: The scrollbar is always displayed regardless of whether the content overflows the viewable area.
Dynamic Show: The scrollbar is displayed when the content overflows the viewable area; hidden when the content does not overflow.
Auto Show: The scrollbar is automatically shown or hidden based on whether the content overflows the viewable area.
Rolling Direction: Scrollable directions when content exceeds the viewable area, including up, down, left, right, vertical, horizontal and arbitrary directions.
Rolling Transfer: Whether the scroll event is propagated to the parent control, disabled by default.
Enable Rolling Inertia: When the content is swiped on the interface and the finger is released, the content continues to scroll for a certain distance with a specific inertia before eventually stopping.
Elastic reel Effect: When scrolling to the edge of the content, the interface offsets slightly beyond the edge and then bounces back to the inside of the viewable area.
Only single child objects are allowed to scroll: Only the currently touched/selected subobject can be scrolled; other subobjects do not scroll along with it.
Hidden: Whether the control is visible in the view.
Alpha: The transparency of the control, ranging from 0 to 255. 0 means fully transparent, and 255 means fully opaque.

4.5. Container widgets
4.5.1 Description of Container Controls
Container controls refer to special components that can contain and manage other controls (such as buttons, text boxes, lists, etc.). Their role is to organize multiple related controls into a logical whole, facilitating layout management, unified operations, or the implementation of specific control behaviors.

In Giraffe IDE, the provided container controls include:

container: The basic container control, mainly used for integrating and managing various elements on the interface. It can hold related controls centrally, making it convenient to uniformly set attributes such as position and style, and serves as the fundamental carrier for building interface structures.

tileview : A container control that supports sliding operations. It can hold multiple sub-content blocks or sub-pages inside, and users can switch to display different content by sliding horizontally or vertically. It is suitable for displaying multiple groups of associated yet independent information within limited interface space. For details, please refer to Chapter 4.29 of the Software API

scrollcont : A container control with circular scrolling functionality. When the internal content exceeds its display range, scrolling to the boundary will seamlessly connect back to the starting part, forming a circular effect. It is often used to continuously display a large amount of sequential or repetitive content, enhancing the coherence of content presentation. For details, please refer to Chapter 4.28 of the Software API

img

Any controls except video controls can be placed inside a container.

Right-clicking the container allows access to the sub-page editing area, facilitating the editing of sub-controls.

img

For sub-pages, you can modify the "Available Control Multiplier", which sets the size of the area where sub-controls can be placed.

4.18. Chart
The chart widget is mainly composed of a table background, horizontal and vertical dividers and data lines.

The data line is a collection of data points, i.e., a collection of y-coordinate points. The range of the y-coordinate can be realized by setting the minimum and maximum values of y on the host computer.

The x-coordinate is calculated as an automatic offset based on the number of data points without user intervention. Offsets are processed in equal parts based on the total number of data points in the data line.

Horizontal spacing between each data point = width of the chart widget / total number of data points

There are several types of chart displays:


Multiple data lines can be plotted simultaneously in a chart. The data lines can be added via the grf_chart_add_series interface and the colour of each data line can be set individually.

Diagram of the widgets:


As can be seen from the diagram above, there are scales on both the x-axis and the y-axis. The scales consist of a main scale, a secondary scale and a scale value, with the long scale being the main scale and the short one being the secondary scale. The contents of the scale value are set with '\n' separating them, such as 0\n10\n20\n30\n40\n50. The scale values on the x-axis are arranged from left to right and the y-axis is arranged from top to bottom. The tick marks marked with red circles on both scales are the last scales on each axis, which must be enabled before they can be displayed.

 Alt text

The size of the position occupied by the label and the scale: This refers to the chart margin value reserved for displaying ticks and value labels. The parameter value must be greater than or equal to the actual size of the label text plus the tick size; otherwise, the labels or ticks will be truncated (displayed incompletely).

4.19. Table
The table control consists of rows and columns that combine to form one cell after another, inside which only text can be stored. The width of each column is equal to the control's width / number of columns, and the height is adaptive, as determined by factors such as the internal font size and the internal margins.

In setting the text content of the cell, when the width of the text content is greater than the width of the column where text content will automatically be transposed, resulting in an increase in the height of the cell, manually let the text content translocation, you can use \n escape character, but the effect of this manual translocation is a little special, it will be transposed between the two paragraphs of text and the cell to add a horizontal horizontal line of equal width, resulting in the cell looks like it has been divided in two.

Cells also have a merge, alignment and other basic table operations, the table in the middle of the row separator is composed of the border of each cell, so the title and content cells must be set to the border to display.

4.20. Spinbox
The spinbox data box widget is essentially a txtbox text box widget, but with extended functionality, mainly used to increment or decrement data.

The diagram of the widget : 

When the value is positive, the + sign is displayed and when the value is negative, the - sign is displayed. The spacing between the sign and the value can be set by the host computer.

The cursor is used to indicate the precision of the current step. When the step is less than 10, the cursor flashes on the single digit, when the step is less than 100, the cursor flashes on the tens digit, and so on. To move the cursor left is to enlarge the step by a factor of 10, to move the cursor right is to reduce the step by a factor of 10.

4.21. GIF
GIF widget can be used in the same way as image widget, just create a GIF widget in the design area, then add the gif you want to display to the resource library, and select the image in the initial gif parameter of the GIF widget.

The GIF widget does not support transparency, that is, the transparency parameter of the widget is invalid.The playback state of the GIF widget is divided into a static state and a loop playback state, when you select the static state, the result will be the initial static state of the GIF picture; when you select the loop playback state, the result will be the complete playback of the GIF. The number of times of playback can be set manually, when the parameter is 0, it means the number of times of playback is not limited.

When the GIF image is decoded, the RAM required in the decoding process is determined by the color depth.

For a color depth of 8, the required RAM is width * height * 3 memory.
For a color depth of 16, the required RAM is width * height * 4 memory.
For a color depth of 32, the required RAM is width * height * 5 memory.
For example, an 800 * 480 GIF image with a color depth of 16 requires 800 * 480 * 4 (approximately equal to 1.4M) RAM.

4.28. ScrollCont
The ScrollCont control supports the creation of multiple sub-pages, and also allows specifying the number of sub-pages that can be displayed in the parent control area (only odd numbers such as 1, 3, 5, 7, etc. are supported).

Multiple sub-pages are presented in a sliding loop manner. During the sliding process, the sub-page in the central position will maintain its original size, while the sub-pages on both sides will be displayed in a reduced size. The reduction ratio is set in the property bar, with a range of (1, 256), where 256 represents the original size.

Regarding the sliding direction of the sub-pages, both horizontal and vertical sliding methods are supported, and users can freely switch according to the actual application scenarios. In addition, the parent control also provides an option for setting the sliding arc radius. By adjusting this parameter, users can control the radian of the sliding track. The larger the radius value, the more obvious the corresponding sliding radian will be.

In terms of sliding effects, the system provides two modes for users to choose: progressive and protrude, so as to meet different visual needs.

 Alt text

Note: The displayed number must be less than the number of child controls and must be an odd number.

7. Timer Task System
Giraffe has a built-in timer system. You can register a timer task which runs a set task callback function at a set period. It is worth noting that software timers have a certain amount of error cannot be used for very precise tasks (e.g. stopwatch, etc.). The timers are non-preemptive, which means that one timer cannot interrupt another timer. Functions from any API can be called in the callback function.

7.1. API
7.1.1. Create Tasks
grf_task_t grf_task_create(grf_task_cb_t task_xcb, u32 period, void* user_data);
task_xcb: task callback function

period: the period of the task callback, in ms, the default is to execute the task callback function periodically at this value

user_data: user-defined data, can be passed to the task callback function, if not used, pass NULL

Return value: returns a pointer to the created task handle, which can later be used to modify its properties

Note: Creating a task does not mean creating a thread. It only means that the display thread will call and execute the task callback function within the set period. Therefore, there is no need to add a delay function in the callback function, and no high-latency function processing is allowed. Otherwise, the screen refresh rate will be reduced.

7.1.2. Delete a Task
void grf_task_del(grf_task_t *task_t);
task_t: task handle pointer

Return value: none

When we don't need a task, we can use this interface to delete it and free up the resources in the heap.

7.1.3. Get the User Parameters for the Task
void grf_task_get_user_data(grf_task_t *task_t);
task_t: task handle pointer

Return value: return the pointer to the obtained user parameter, or NULL if no user parameter is set.

7.1.4. Change the Task Callback Function
void grf_task_set_cb(grf_task_t *task_t, grf_task_cb_t task_cb);
task_t: task handle pointer

task_cb: task callback function

Return value: none

This interface is used when a project needs to dynamically modify callback functions, but is not normally used.

7.1.5. Set the Task Callback Period
void grf_task_set_period(grf_task_t *task_t, u32 period);
task_t: task handle pointer

period: period of the task callback, in ms

Return value: none

This interface is used when a project needs to dynamically modify the period of a task callback function, but is not normally used.

7.1.6. No Waiting Execute Immediately
void grf_task_ready(grf_task_t * task_t);
task_t: task handle pointer

Return value: none

This API allows the called task callback function to be run immediately, without having to wait for its first run cycle. Note that this means that the first run cycle does not have to wait, but the subsequent run cycles do.

7.1.7. Make the Task Callback Function Run Only Once
void grf_task_once(grf_task_t *task_t);
task_t: task handle pointer

Return value: none

This API enables the called task callback function to be run only once, instead of periodically. After a task call has finished running once, the task system will automatically delete this task internally via grf_task_del.

7.1.8. Reset a Task
void grf_task_reset(grf_task_t * task_t);
task_t: task handle pointer

Return value: none

This API can make the reset task's waiting period set to 0. For example, if the callback period of task 1 is 1000ms, and it has already waited for 400ms, and normally it needs to wait for 600ms for the callback function of task 1 to be run, if you use grf_task_reset function to reset the task 1, then the waiting period of 400ms will be invalidated, and the task will need to wait for a full period of 1000ms before it will be run again.

7.1.9. Set task pause
void grf_task_set_cb_pause(grf_task_t *task_t);
task_t: task handle pointer

Return value: none

This API can pause a running task, but does not delete it.

7.1.10. Resume the task
void grf_task_set_cb_resume(grf_task_t *task_t);
task_t: task handle pointer

Return value: none

This API allows a suspended task to resume running.

8. Introduction to Resources and How to Add them
Resources are mainly divided into font, image, audio and video resources, etc.

8.1. Memory Resource
In order to achieve consistency of use, the same Giraffe project can be directly adapted to different hardware platforms, but the memory varies from one hardware platform to another.

When the interface is displayed, it needs to dynamically request and release User Memory (after removing the memory needed for the system and driver display to run, the remaining memory is user memory).

User memory varies by platform, TR660-10MB, TR540/TR240-4MB, TR230-1MB(Different hardware boards, different hardware functions, user memory may be a little different, you can check the remaining memory through the debugging information), Among them, TR540 and TR230 have at least 300KB left for system and driver to call dynamically, and TR660 (linux) has at least 1MB left.

8.1.1. User Memory Usage Distribution
System and driver: After the normal operation of the program, the system and driver will also be dynamically applied and released, but the larger memory power-up has been applied, the program needs to be dynamically applied and released after the normal operation of very little;

User programs (code, global variables, static variables) that are automatically requested when powering up and running and are not released when running;

Views and controls: Global controls are requested when they need to be displayed and are not released when switching views. Private controls are requested when they need to be displayed and are released by switching views;

Font Display: Requested when character is displayed, not released. The amount of memory requested depends on the number of fonts and font sizes used, e.g., if a font is used in size 16 and 24, it will take up memory twice. Multiple labels use the same font and font size and are only applied once.

8.1.2. Memory Optimization Method
User programs reduce the amount of code written and reduce the use of global and static variables
Do not use global controls unless necessary, use private controls.
Reduction in the number of character sets used
8.1.3. Memory Viewing Method
Giraffe IDE -> Change Project Propertiy -> Parameter display -> Used memory(only on windows) -> ok -> Memory displayed on the Run window
Get memory usage by calling grf_mem_get_free() and grf_mem_get_used()
Note: The windows run page supports viewing used memory, the microcontroller side does not. Since windows memory is large enough for the Giraffe application, there is no need to care about the remaining memory, only the memory already used by the application. The microcontroller also uses this memory dynamically, so the virtual window view of used memory can be used as a pre-determination of memory usage on the microcontroller side. On the microcontroller side the driver layer and the system also use memory dynamically and need to keep an eye on the remaining memory.

Memory Used It only counts the memory usage of views and controls when they are dynamically created and exited, and is used to pre-determine the memory usage when debugging on the PC side.
Memory Used does not count the use of global variables in the user code, and too much use of global variables will reduce the remaining memory on the microcontroller side.
Memory Used does not count the use of local variables in the user's code, showing that the thread stack reserves enough theory to run the program, users try to avoid the use of large temporary variables in function. If you do need large arrays, you can use static variables, global variables, or grf_malloc heap requests (grf_free releases them after use).
Since the data is displayed on the top layer, but the top layer may be used by the customer and overwrite it, it can be viewed by debugging the print (1s print).

9. Modbus RTU API
Introduce the usage of Giraffe Modbus function.

9.1. Running Process
As the Master execute the process:

Since Modbus RTU interact between two devices through the UART serial port, the Master needs to wait for the response from the other device after sending data. If waiting directly in the display thread, it will reduce the fluency of the display UI. Therefore, when initializing the Master, a command execution thread is created to avoid affecting the UI display during the waiting process.

Note: The command execution thread has a higher priority than the display thread. Therefore, after sending a command, the command will be executed immediately. The execution result will be stored in the result FIFO. If a large number of command are executed within one display refresh cycle for user command, the result FIFO may become full, resulting in the loss of subsequent result data.

img

As the Slave execute the process:

The slave machine accept command thread during initialization, which is used to receive command from the Master and respond immediately to avoid the Master waiting for a response timeout.

img

Note: Since the execution result of command are cached through FIFO, the actual reception of command results and execution of corresponding UI actions in the display thread. Therefore, if a large number of command are executed within one display refresh cycle for user command, the result FIFO may become full, resulting in the loss of subsequent result data.

#define HEAD_FH 0x5A
#define HEAD_FL 0xA5
#define REG_LEN 0x800
static u16 ctrlreg[REG_LEN] = {0};

//Set register data
//addr:register address to be set
//data:data to be set
s32 grf_reg_set(u16 addr,u16 data)
{
    if(addr>REG_LEN){
        return GRF_FAIL;
    }
    ctrlreg[addr] = data;
    // grf_printf("set reg %04X = %04X\n",addr,data);
    return GRF_OK;
}

//Get the register data
//addr:register address to be fetched
s32 grf_reg_get(u16 addr)
{
    if(addr>REG_LEN){
        return GRF_FAIL;
    }
    // grf_printf("get reg %04X = %04X\n",addr,ctrlreg[addr]);
    return ctrlreg[addr];
}
//*********write reg********** 
//TX - FH FL len cmd addr data0 data1
//     5A A5 07  82  0001 FFFF  FFFF
//RX - NULL

//*********read reg********** 
//TX -  FH FL len cmd addr reglen
//      5A A5 04  83  0001  02
//RX -  FH FL len cmd addr reglen data0 data1
//      5A A5 08  83  0001  02    FFFF  FFFF

static s32 grf_reg_s_set(u16 addr,u8* data,u8 len)
{
    if(addr>REG_LEN){
        return GRF_FAIL;
    }
    u8 i=0;
    for(i=0;i<len;i++){
        ctrlreg[addr+i] = (data[i*2]<<8)+data[i*2+1];
        // grf_printf("reg_s_set %04X = %04X\n",addr+i,ctrlreg[addr+i]);
        grf_reg_set_user(addr,ctrlreg+addr+i,1);
    }
    return GRF_OK;
}

//Register send-function
//addr:register address to be sent
//len:data length to be sent
s32 grf_reg_com_send(u16 addr,u16 len)
{
    u8 txdata[257] = {0};
    u32 i=0,j=0;
    txdata[i++] = HEAD_FH;
    txdata[i++] = HEAD_FL;
    txdata[i++] = len*2+4;
    txdata[i++] = 0x83;
    txdata[i++] = addr>>8;
    txdata[i++] = (addr&0x00ff);
    txdata[i++] = len;
    for(j=0;j<len;j++){
        txdata[i++] = (ctrlreg[addr+j]>>8);
        txdata[i++] = (ctrlreg[addr+j]&0x00ff);
    }
    grf_drv_uart_send(drv_uart,txdata,i);
}

//Instruction parsing function
static s32 grf_comm_handle(u8* data)
{
	u8 cmd = data[3];
	u8 len = data[2];
    u16 addr = (data[4]<<8)+data[5];
    u32 i=0;
    if(addr>REG_LEN){
        return GRF_FAIL;
    }
    //    grf_printf("cmd=%x\n",cmd);
    switch (cmd)
    {
        case 0x82: //write registers
        {
            u32 regcount = (len-3)>>1;
            grf_reg_s_set(addr,data+6,regcount);
        }
        break;
        case 0x83: //read registers
        	grf_reg_com_send(addr,data[6]);
        break;
    }
    return GRF_OK;
}

#define UART_LASTBUFF 1
#if UART_LASTBUFF
#define RX_BUF_LEN   1024
static u8 RX_HAND_BUF[RX_BUF_LEN];
#endif

//Serial port receive interrupt service function
//databuf:received data
//datalen:length of received data
static void recive_data_handle(u8* databuf,u32 datalen)
{
	u16 i=0;
	static u16 last_data_num=0;
#if UART_LASTBUFF
	if(last_data_num + datalen > RX_BUF_LEN){
		datalen = datalen-last_data_num;
	}
	if(last_data_num){ 
		memcpy(RX_HAND_BUF+last_data_num,databuf,datalen);
		databuf=RX_HAND_BUF;
		datalen+=last_data_num;
		last_data_num = 0;
	}
#endif
#if 1
    grf_printf("uart rx[%d]:",datalen);
	for(i=0;i<datalen;i++){
		grf_printf(" %02X",databuf[i]);
	}
	grf_printf("\n",datalen);
#endif
	if(datalen>=6){//Shortest command consists of 6 characters
		for(i=0;i<=datalen-6;i++)
        {
			if((databuf[i]==HEAD_FH) && (databuf[i+1]==HEAD_FL))
			{
    			if(databuf[i+2] <= (datalen-i-3)){
                    if(grf_comm_handle(databuf+i)==GRF_OK){
                        i += (databuf[i+2]+3)-1;
                    }
    			}
		        else{
                    last_data_num = datalen-i;
                    break;
		        }
			}
		}
	}
	else{
		last_data_num = datalen;
	}
#if UART_LASTBUFF
	if(last_data_num<=256 && last_data_num!=0)
	{
		u8  last_data_buf[256] = {0};
		memcpy(last_data_buf,databuf+i,last_data_num);
		memcpy(RX_HAND_BUF,last_data_buf,last_data_num);
	}else{
		last_data_num=0;		
	}
#else
	last_data_num = 0;
#endif
}

typedef enum {
    KEY_UP,
    KEY_DOWN,

    KEY_DOWN_LONG,
    KEY_NO
}key_state_e;
typedef struct  
{
    key_state_e key_state;
    u32 key_time;
    
    key_state_e last_key_state;
}key_param_t;
key_param_t key_para[2] = {0};
void irq_task_test1(grf_gpio_irq_t *irq_t)
{
    u8 irq_pin = grf_gpio_group_get_irq_pin(irq_t);
    u8 irq_mode = grf_gpio_group_get_irq_mode(irq_t);
    // grf_printf("irq_pin = %d irq_mode=%d \n",irq_pin,irq_mode);
    switch (irq_pin)
    {
        case 0://PE 0 IRQ interrupt
            if(irq_mode == GRF_GPIO_IRQ_RISING){
                key_para[0].key_state = KEY_DOWN;
                key_para[0].key_time = grf_sys_time_ms();
            }
            else{
                key_para[0].key_state = KEY_UP;
                key_para[0].key_time = grf_sys_time_ms();
            }
        break;
        case 1://PE 1 IRQ interrupt
            if(irq_mode == GRF_GPIO_IRQ_RISING){
                key_para[1].key_state = KEY_DOWN;
                key_para[1].key_time = grf_sys_time_ms();
            }
            else{
                key_para[1].key_state = KEY_UP;
                key_para[1].key_time = grf_sys_time_ms();
            }
        break;
    
    default:
        break;
    }
}
static key_state_e key_get_state(key_param_t *key_para)
{
    key_state_e state_e = KEY_NO;
    if(key_para->key_state == KEY_DOWN){
        u32 cur_time = grf_sys_time_ms();
        u32 key_time = key_para->key_time;
        if(cur_time>key_time && cur_time-key_time>10){ //10ms jitter
            key_para->last_key_state = KEY_DOWN;
            state_e = KEY_DOWN;
            if(cur_time-key_time>2000){//2s long press
                state_e = KEY_DOWN_LONG;
            }
        }
    }
    else{
        if(key_para->key_state == KEY_UP && key_para->last_key_state == KEY_DOWN){
            state_e = KEY_UP;
            key_para->last_key_state = KEY_UP;
        }
    }
    return state_e;
}
static void irq_task_cb_t(grf_task_t *task_t)
{
    key_state_e state_e = key_get_state(&key_para[0]);
    if(state_e!=KEY_NO){
        grf_printf("KEY 0 state_e=%d \n",state_e);
    }
    state_e = key_get_state(&key_para[1]);
    if(state_e!=KEY_NO){
        grf_printf("KEY 1 state_e=%d \n",state_e);
    }
}

/*
*Function: u32 grf_flash_write_Data(__u8 type,u32 addr,__u8* databuf,u32 len)
*Description: write data to flash(grf_data.bin)
*Formal parameter:
	type - Storage device type,0: flash 1: TF card 2: U disk
	addr - Write first address of data (0-0xFFFFFFFF)	
	databuf - Data buffer to be stored for writing
	len - Write data length in bytes
*Return: Number of bytes actually read
*/
u32 flash_write_Data(u8 type,u32 addr,u8* databuf,u32 len)
{	
    char wfilename[256] = {0};
	u8* pathtype = NULL;
	s8 ret = 0;
	u32 reallen = 0;

	//Select a path
	if(type==1){
        pathtype = grf_dir_get_ddisk();
        ConnectStr(wfilename,256,pathtype,BINFILE);               
	}
	else if(type==2){
        pathtype = grf_dir_get_tfdisk();
        ConnectStr(wfilename,256,pathtype,BINFILE); 
	}
	else{
        pathtype = grf_dir_get_udisk();
        ConnectStr(wfilename,256,pathtype,BINFILE);
	}
	// grf_printf("wfilename=%s\n",wfilename);
	if(wfilename[0]!=0)
	{
		FILE*   datafp=NULL;
		datafp = fopen(wfilename,"r+b");
		if(datafp == NULL){
        //First think that the file has not been created
			datafp = fopen(wfilename,"wb");	
		}
		if(datafp==NULL)
		{
			printf("create grf_data.bin fail\n");
			fclose(datafp);
	        goto __err;
		}
		ret = fseek(datafp,addr,SEEK_SET);
		//grf_printf("ret=%d\n",ret);
		if(ret!=0){//If the address to be written exceeds the total size of the file, first expand the file with 0, and then write the data.
			u32 endlen,offset;
			void *data_p;
			fseek(datafp,0,SEEK_END);
			endlen = ftell(datafp);
			offset = addr-endlen;
			data_p = malloc(offset);
			memset(data_p,0, offset);
			fwrite(data_p,1,offset,datafp);
			free(data_p);
			//grf_printf("endlen=%d offset=%d \n",endlen,offset);
		}
		reallen = fwrite(databuf,1,len,datafp);
		fclose(datafp);
	}
__err:
	return reallen;	
}

/*
*Function: u32 grf_flash_read_Data(__u8 type,u32 addr,__u8* databuf,u32 len)
*Description: read data from flash(grf_data.bin)
*Formal parameter:
	type - Storage device type,0: flash 1: TF card 2: U disk
	addr - Read first address of data (0-0xFFFFFFFF)
	databuf - Data buffer to be stored for reading
	len - Read data length in bytes
*Return: Number of bytes actually write
*/
u32 flash_read_Data(u8 type,u32 addr,u8* databuf,u32 len)
{
    char rfilename[256];
	s8 ret = 0;
	u32 reallen = 0;
    memset(rfilename,0,256);	
	//Select a path
	if(type==1){
        u8* pathtype = grf_dir_get_ddisk();
        ConnectStr(rfilename,64,pathtype,BINFILE);               
	}
	else if(type==2){
        u8* pathtype = grf_dir_get_tfdisk();
        ConnectStr(rfilename,64,pathtype,BINFILE); 
	}
	else{
        u8* pathtype = grf_dir_get_udisk();
        ConnectStr(rfilename,64,pathtype,BINFILE);
	}
	// grf_printf("rfilename=%s\n",rfilename);
	{
		FILE*   datafp=NULL;
		datafp = fopen(rfilename,"rb");
		if(datafp==NULL)
		{
			grf_printf("open grf_data.bin fail\n");
			fclose(datafp);
	        goto __err;
		}
		ret =fseek(datafp,addr,SEEK_SET);
		if(ret==0){//Addressing success
			reallen = fread(databuf,1,len,datafp);
		}
		else{
			grf_printf("The location exceeds the file size,addr=0x%04x\n",addr);
		}
		fclose(datafp);
	}
__err:	
	return reallen;
}

static void prase_data(grf_can_msg_t *msg_t)
{
	if(msg_t->dlc == 0)

		grf_printf("%s error data must > 0\n",__func__);
		return ;
	}

	for(int i = 0;i < msg_t->dlc;i++)
	{
		grf_printf("%02X ",msg_t->data[i]);
	}

#define CAN_ID 0x55

HMI Instruction Set Properties

HMI Enable : Enables or disables the entire instruction set communication function; it is disabled by default.

PC analog Port: A virtual serial port used for simulation debugging on the PC, which only takes effect when running in PC simulation mode.

User Port: The physical serial port number used during actual hardware operation. -1 indicates the default port; any other value specifies a custom port number.

Baud: The serial communication rate.

Byte Size The number of valid binary bits contained in each frame of data. The available options are BIT_8 and BIT_7.

Parity Flag: The check method. Available options: No parity, Odd parity, Even parity.

Stop Bit: The frame end flag bit, usually 1 bit.

Frame Header High Byte (Hex): The high 8 bits of the communication frame start flag, used for frame synchronization.

Frame Header Low Byte (Hex): The low 8 bits of the communication frame start flag, which together with the high byte form the complete frame header.

Buffer Length: The size of the transmit/receive data buffer (unit: bytes).

Timeout (ms): The maximum time to wait for a response from the lower computer. If the timeout is exceeded, the communication is deemed to have failed.

Write Command Return Report: When checked, the upper computer will wait for and verify the return result from the lower computer when issuing write commands, ensuring successful execution.

CRC Enable: When enabled, a CRC check code is appended to the communication frame to detect errors during data transmission and improve communication reliability.

Event Dispatch Mode: Used to configure the strategy for reporting control events to the lower computer.

0 None：: No control events are reported; only basic communication is performed.

1 All Controls: Events generated by all controls are reported to the lower computer. Click the "Configure Event Commands" button on the right to open the event configuration window, where you can precisely select the event types to be reported.

2 Self Controls: Only event types configured via "Select Event Commands to Deliver" for individual controls are reported.

When "2 Self Controls" mode is selected,event reporting is no longer a global configuration, and precise settings must be made in the property interface of each individual control:

In the Special Config category of the control properties, find the Dispatch event commend selection option and click the Click Configure button.


Follow Triggerable Events: When this option is checked (default), the event types reported by the control to the lower computer will automatically be fully consistent with those selected in "Triggerable Events". If the selection in To Select Events is modified, the configuration of Dispatch event commend selection will be automatically updated synchronously.
After uncheckingFollow Triggerable Events, you can manually select the event types to be reported. For example, a control can respond to both "Control Pressed" and "Control Touch Lifted" events, but you can choose to report only "Control Touch Lifted" to reduce unnecessary communication traffic.

Data Length
If the highest bit of the first byte is 1, i.e. 0x80, it means that the parameter is 2 bytes, otherwise it is 1 byte (1-128).

Read/Write Instruction
The instruction reads and writes in the form of a parameter, 11(read)/22(write)

Read data:
TX: Frame Header + Data Length + 11 + Instruction Type + Control Command + (CRC16)

RX: Frame Header + Data Length + 11 + Instruction Type + Control Command + Data + (CRC16)

Write data:
TX: Frame Header + Data Length + 22 + Instruction Type + Control Command + Data + (CRC16)

RX: Frame Header + Data Length + 22 + Instruction Type + Control Command + 00/01 + (CRC16)

Notes:In received data, 00 indicates a successful write operation; non-zero values indicate failure.

Examples:

Read Driver Version

TX : 5A A5 04 11 B0 00 02
RX (Success): 5A A5 0A 11 B0 00 02 31 2E 34 2E 35 00 // Version: 1.3.5
RX (Failure): 5A A5 04 11 B0 00 02 // Returns original command with no data.

Write Backlight Brightness

TX: 5A A5 04 22 B0 02 32 // Set brightness to 50
RX: 5A A5 04 22 B0 02 00 // 00 = success, non-zero = failure. Note: This response can be disabled.

CRC16 Sample Code
u16 crc16_ccitt(u8 * tmp, u32 num)
{
    u16 CrcTmp = 0x0;
    u32  i = 0, j = 0;
    u16 s = 0;
    u16 tmpcrc = 0;

    for (i = 0; i < num; i++)
    {
        tmpcrc  = tmp[i] << 8;
        tmpcrc  &= 0xff00;
        CrcTmp  ^= tmpcrc;

        for (j = 0; j < 8; j++)
        {
            s = CrcTmp & 0x8000;
            CrcTmp <<= 1;
            CrcTmp  &= ~(0x0001);

            if(s){
                CrcTmp ^= 0x1021;
            }
        }
    }

    return CrcTmp;
}

Parameter Description Rules
A single instruction may include multiple parameters, described as param1, param2, etc.
The parameter type is specified in the first parenthesis after param. For example, param1(string) indicates that parameter 1 is of type string, conforming to the byte count and data range defined in the parameter type table.
The data range is determined by the parameter type. However, some parameters may have a smaller range than the type allows. This is specified in the second parenthesis. For example, param1(u8)(0-100) indicates that parameter 1 is of type u8 with a range of 0-100. If no second parenthesis is provided, the range defaults to the type's full range.
After describing the type and range, the parameter's functionality is explained after a colon.
Commands are sent and received in the order of parameters' numbering.
